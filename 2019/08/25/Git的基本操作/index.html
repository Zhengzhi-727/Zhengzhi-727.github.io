Git的基本操作

发表于 2019-08-25 分类于 代码世界

集中式版本控制

其实在 Git 之前，很多人用的是 CVS ，也就是 Centralized Version Control ，这里的 Centralized 就是集中的意思。也就是那会，经过血的教训，人们开始意识到版本控制的重要性了，然后就搞了台服务器，在这台服务器里面就专门来管理各种版本，保存每个版本的更新、修改等操作记录。这时候一遇到什么问题，就可以直接去这台服务器找，比如你误删了代码，就到这台服务器找之前的版本，回退一下就可以了。

你们多人协作的时候，每次完成代码就更新到这台服务器上面去，这台服务器都完整的记录了你们的每个版本，每个动作。妈妈再也不用担心别人骂我傻逼了。

是不是觉得还挺不错的呢，但是这种集中式的版本控制方式有一个弊端，就是哪一天服务器突然傻逼了，那么你就没办法去上面拿到别人提交的版本，你也没办法更新你最新的版本上去。更可怕的是，如果哪天这台服务器直接坏掉，挖槽，那你们之前那些所有的版本记录不就烟消云散了吗？如果没人备份的话，那岂不是操了个操了？？

分布式版本控制（Git）

为了解决这个问题，牛逼的人就想到了一个方法，他们在想，既然这样，那么何必不在每个人的电脑里面，都装一份服务器上面的版本控制管理的镜像呢？也就是说，除了服务器上面有之外，我们的电脑，也有一份一样的控制系统。这个时候如果说服务器傻逼了，那么我们的内心毫无波动，还可以顺便唱一句：“怎么大风越荡，我心越浪”了，因为每个人本地都有（你每一次对版本的更新或者拉取，你的控制系统就是和服务器上的一样最新的，所以不用担心）啊，直接恢复就可以了。

git的基本操作（不同操作系统命令有细微不同）

git –version：获取版本号

mkdir games：创建文件夹

gamesgit init：初始化。创建一个空仓库，生成.git隐藏文件，实现Git管理必备文件ll查看当前文件下的所有文件

git status：查看状态

touch test.txt：新建test.txt文件

git add test.txt：添加test.txt文件到 track，用git status可查看待提交的文件（ 把你工作区里面的新修改的文件生标记一下追踪，然后生成快照，记录起来，放到暂存区里面。这个时候并不是直接提交就到 Git 仓库里面去。）

git add . ：添加当前目录下的所有文件到track

git commit -m “XXX”：提交上一步add的文件给Git管理。m后面必须说明提交注释

git help：列出所有git命令的解释

git –help init：查看init的用法

git log：查看日志

git log -5：查看最后5条日志

git commit –amend：修改上一次提交的信息（o切换编辑，shift+zz退出）

git reset test6.txt：撤销track中的test6.txt文件（将test6从暂存区中移除）

git reset . ：撤销track中的所有文件（将所有文件从暂存区中移除）

git reset –soft HEAD~：撤销上一步的提交，回到上一次add后的状态

git reset –hard +id号：退回到指定版本（id号位数输入到可以区分不同版本的长度即可）

git branch显示项目分支情况

git branch feature_1：创建feature_1分支（在master下新建分支）

git checkout feature_1：切换到feature_1分支（在master下新建分支）

git checkout -b feature_2：创建并切换到feature_2分支（在master下新建分支）

vim test.txt：编辑test.txt文件

git merge feature_1：合并feature_1分支（在master下合并分支）

git branch -d feature_1：删除feature_1分支

git stash：暂存，当写到一半code时需要切换分支时需将当前分支的code暂存才能成功切换，否者只能先提交才能切换分支

git stash apply：恢复暂存的code

git stash apply stash@{1}：如果暂存多次，可通过git stash list来查看，这里选择了恢复1号暂存
